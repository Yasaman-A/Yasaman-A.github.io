In this activity, you will work with some git commands and see how they can affect the git history. More specifically, you will practice integrating changes in a pull request into your main branch using three methods of merging, squash and merging, and rebasing.

Collaboration format - You should work in groups of 3. One submission with the name of all members is required. If for any reason you are unable to work in a group, you must do all the steps by yourself. This will require you to create three repositories and integrate changes with three different ways of merging, squash and rebasing.
  1. Create a private github repository. You should have a main branch as your default branch. Assume that there are some issues in your repository. You can create an issue from your issue tab.

  1. The common steps should be done by all members of the team. To make the comparison easier, all members of the team should try to use the same file names and commit messages in the following steps.
  1. Clone this repository to your computer and start working on issue 1.
  1. First, update your main branch by pulling it from the remote main branch.
  1. Then, branch off from the latest version of main and create a new issue branch, e.g., issue1. Create a file, a.txt, with a content of your choice in this branch. Then, add and commit this file.
  1. Your task in issue 1 requires more implementation. Create another file, aa.txt, with a content of your choice in this branch. Then, add and commit this file.
  1. Assuming your task is done for issue 1. Now, it is the time for submitting a pull request. Push your changes to remote repository in a branch with the same name as your local branch.
  1. Now, it is the time for merging your pull request. Usually, another member of the team should review the pull request before merging. We skip this step for simplicity and jump to merging.
  1. You will do this in three ways. If you are working in groups of three, each member can do it in one way and then you can compare your results.
  1. Team member ONE - you should merge your pull request into the main branch. Then, you should switch to your local main and update it with the latest changes in the remote main.
  1. Team member TWO - you should squash and merge your pull request into the main branch. Then, you should switch to your local main and update it with the latest changes in the remote main.
  1. Team member THREE - you should rebase and merge your pull request into the main branch. Then, you should switch to your local main and update it with the latest changes in the remote main.
  1. Review the git logs and visualize the history. You should do this for all three ways of merging and include the results in your submission. Using the git log command, review the log history. From the log history, try to manually visualize the git history. You may draw it using pen and paper, or use online tools like draw.io. To show the commits you can use their SHA or you can name them as C1, C2, ... .
  1. Search and find a git visualization tool and use it for visualizing your git history. For example, if you are using VS Code, you can use the Git Graph plugin to visualize your git history. Compare the result of the visualization tool with your own visualization. If needed, make corrections to your drawing.

  1. Compare the history logs and the visualizations created by the three different merging techniques you used. You may consider the following aspects for your comparison:
    • How many commits are added to the main branch as a result of merging your issue1 branch into the main branch?
    • Who is the author of the new commits?
    • Is there a link between the new commit to its parent commit from the issue1 branch?
  1. Based on your comparison, summarize the differences between the three ways of merging that you tried, and include this summary in your report.
  1. Optionally, you may search and read about these three merging techniques and use your findings to complete your answer.
  1. You finished your task. One submission per group (group of three) is required. Your submission should include:
    • visualizations of the git history for the three different techniques you compared.
    • summary of your comparisons of these three merging techniques.
